摘要 — 推荐范围

构建一个面向单用户的 Web 端"DrawCopilot 演示工作台",基于客户自有数位屏硬件,实现以下三件事:

F1 · CEO 最爱的演示亮点

"一键润色我的画" — Sketch-to-Image 风格化

用户随手涂鸦 → AI 返回 4 种风格、共 4 个润色版本。这就是 CEO 想看的"AutoDraw + 三星 Gemini"瞬间。

F2 · 真正的战略楔子

笔触过程记录器 + UIM 标准导出

以全保真度采集每一笔事件 (x, y, p, tilt_x, tilt_y, t),同时导出 JSONL 与 Wacom Universal Ink Model。数据飞轮的"种子粮"。

F3 · 不依赖 ML 的过程感知功能

笔触回放 + 透视 / 对称辅助

纯几何计算实现的功能,证明今天就能让丰富的笔数据立刻产出可见价值——不需要任何新模型。

F4 · 仅作余量目标

Sketch-RNN 幽灵笔触(开箱即用)

仅当 F1–F3 在第 9 天前全部稳定后才上线。用于验证 Copilot 式交互手感,演示时必须坦白说明:这是 2017 年的公开模型,不是"我们专属"的。

明确推迟:客户 PPT 里描述的、基于压感/倾角/时间的"下一笔预测"新模型。其 R&D 风险无法塞进两周窗口。详见第 5 节。

1 · 决策问题

两周的范围被三股力同时拉扯:

压力来源含义
快速拿出能打动 CEO 的 Demo客户 CEO 喜欢 AutoDraw 与三星 Galaxy AI 的 Drawing Assist / Sketch-to-Image,并明确告诉我们"这种东西可以很快做出来。"使用现成的图像生成 API(ControlNet-Scribble / SDXL / fal.ai / Replicate / Azure OpenAI Image),数日可上线。
证明真正的战略楔子客户 PPT 第 4、7、12 页:护城河是笔触过程数据,不是被润色后的图片。他们的核心价值是数据入口——以 200 Hz 采集的 (x, y, p, tilt, t) 加完整会话上下文。必须把数据管线本身做出来,不能只是去调别人家的图像生成 API。
尊重团队对 R&D 的现实判断产品负责人备忘:"R&D 周期更长;新增压感/时间/倾角维度的模型需要反复试错……先做一个 1–2 个月内能上线的东西,新模型放到长期。"推迟新的序列建模 ML 工作。两周窗口内只用现成模型。
MVP 必须在 14 天内同时满足这三股力。推荐范围的解法是把"对外可见的演示"与"对内的战略基础设施"解耦,同时确保战略基础设施本身至少能产出一个可见的 Demo(透视辅助 + 回放)。

2 · 推荐功能集

F1 — "一键润色我的画"(CEO 演示亮点)

  • 用户体验:在画布上随手画 → 点击润色 → 弹出 2–4 种风格(水彩、动漫/漫画、概念设计、铅笔速写)。每种风格出 1–2 个变体,用户挑一个变体作为新图层留下。
  • 实现方式:调用现成的、以涂鸦为条件的图像生成模型——通过 fal.ai 或 Replicate 接 ControlNet-Scribble / SDXL-Turbo;备选 Azure OpenAI Image。仅服务端实现。
  • 为什么两周内安全:零新增 ML 工作。集成约 1 天,风格预设的 UI 约 2 天。
  • 注意事项:这是名单里差异化最弱的一项——现在每部消费级手机都有了。我们要把它定位为"对演示友好的外衣";真正的护城河是 F2。

F2 — 笔触过程记录器 + 标准化导出(战略楔子)

  • 用户体验:角落里一个小小的"录制中"指示器。会话结束后出现下载会话日志按钮。
  • 底层在做什么:每一个 PointerEvent 都被写入有类型的事件流——pressure、tiltX/Y、由此推导的 azimuth(方位角)与 瞬时速度,加上会话上下文(图层 id、工具 id、撤销/重做 id;若用户授权还可标注"意图阶段"标签)。
  • 两种导出格式:
    • session.jsonl — 我们自己的逐行 JSON 格式,便于 ML 直接吃。
    • session.uim — Wacom Universal Ink Model(Protocol Buffers v3 + RIFF)。直接呼应客户 PPT 第 9 页的"扩展 WILL / Wintab / HID"。使用 Wacom 已公开的 UIM SDK,不必自创格式。
  • 为什么两周内安全:浏览器的 PointerEvents API 已经把所有字段端给我们了,我们只需把它们落盘;UIM SDK 替我们做二进制序列化。
  • "伪装中的护城河":每一个被录制下的会话,都是我们自己拥有的训练数据。即使长期模型要做 6 个月,等到我们要训练时,也已经积累了 6 个月的纯净一手数据。

F3 — 笔触回放 + 透视/对称辅助(过程感知,无需 ML)

  • 笔触回放:拖动时间轴;画布按原速(或 4 倍)逐笔重绘。Demo 时用来证明"我们拿到的是整个过程,不只是最终那张图。"
  • 透视网格:1/2/3 点透视叠加层;笔的倾角自动对齐当前的灭线方向。立刻就用上了 tilt 数据——纯几何,无模型。
  • 对称镜像:垂直/径向镜像。镜像侧的笔触会以正确的压感与倾角反射——一个不显眼但很能展示"丰富数据为什么有用"的细节。
  • 为什么三件事一起做:它们共同证明"丰富的笔数据 → 可见价值"在没有新模型的前提下也成立。三者都对应客户 PPT 第 10 页:"驱动级智能笔——透视/对称辅助;可教学/可复盘"。

F4 — 仅作余量目标:开箱即用的下一笔幽灵

通过 TF.js 接入 Sketch-RNN(Magenta),用功能开关隐藏。仅当 F1–F3 在第 9 天前都达标才上线。Demo 时必须诚实说明:这是 2017 年公开的、在 QuickDraw 上训练的模型——不是我们将来基于客户的压感/倾角数据所要构建的那个模型。它的工作是验证 UX,不是验证 IP。

本次两周 MVP 明确不做的内容

不做的内容原因
多人协作 / 共享房间(Drawly 风格)不在 CEO 的关注列表;CRDT / WebSocket 复杂度会吞掉整个 Sprint。
语音陪伴(Realtime API)客户没有提;在第一天的成本讨论里就会爆预算。
游戏化课程(ArtWorkout 风格)大量内容 + 产品工作量,客户也没要求。
新的压感/倾角/时间下一笔预测模型是真正的 R&D —— 而它需要的训练数据,正是 F2 上线后才开始产生。
iPad / 移动端原生壳先做 Web Demo;客户的硬件是 Windows 端的数位屏。
账号、付费、分享Demo 在一台预装客户硬件的笔记本上本地运行;Local-First 足矣。

3 · 逐日排期(10 个工作日 / 14 个自然日)

假设小型团队配比:2 名前端、1 名后端/胶水、1 名 ML/集成、1 名设计师(兼职)。如果是单人或结对开发,数量需重新评估;请在第 0 天提出。

日工作流 A — 画布与记录器工作流 B — 润色 API 与界面工作流 C — 回放与辅助
D1锁定范围、建仓、CI;在开发机上把客户硬件接入。验证:在客户驱动下,PointerEvents 是否真的暴露 pressure 与 tiltX/Y——这是第 1 天必须先排掉的头号风险。选型实验:在 fal.ai、Replicate、Azure OpenAI Image 三家中挑选,用 20 张样图测延迟与成本。回放滑块、透视叠加、对称镜像的线框图。
D2矢量画布(Konva + perfect-freehand);笔事件流架构。锁定 4 种风格预设;Prompt 模板。透视网格(1/2/3 点)与对称变换的数学。
D3本地笔触日志 → 落盘 JSONL;基础校验查看器。"涂鸦 → 润色图"端到端首通。回放滑块 MVP(只支持向前)。
D4集成 Wacom UIM SDK,导出 .uim 文件。UI:风格选择网格、生成按钮、结果灯箱。透视叠加层在画布上渲染。
D5会话上下文:图层 id、工具 id、撤销/重做 id、意图阶段标签。变体的拒绝/作为新图层接受流程。对称镜像,确保镜像侧的压感与倾角正确反射。
D6记录器自身遥测:事件每秒数、日志大小、丢失事件。风格 A/B 测试脚手架。4 倍速回放 / 双向时间轴拖动。
D7Sprint 中期内部 Demo —— 团队自己当画师画一小时,看记录器到底记了什么。补漏。
D8余量:Sketch-RNN 幽灵笔触(F4)藏在功能开关后,仅在 A/B/C 全绿时才接入。根据 D7 的 Demo 反馈,微调风格 Prompt。透视 + 对称的 UX 打磨。
D9Demo 数据集:5 段预录会话,导出为 .uim。Demo 备份:4 张涂鸦 × 4 种风格预先缓存,以便离线回放。Demo 用回放片段。
D10端到端冒烟 + 压力测试;为 Demo 准备遥测仪表板。撰写 Demo 脚本。缓冲 / 修 Bug。
D11–D12缓冲日。第 12 天彩排。
D13与内部相关方进行 Demo 排练。锁版。
D14正式向客户演示。
每日站会的纪律:每次站会都必须回答一个问题——"记录器是否仍在以全保真度捕获每一个事件?"这是唯一不可妥协的指标,因为它定义了我们到底有没有真的在采集"护城河数据"。

4 · 技术栈 — 两周 Sprint 的保守选型

每一项决策都要"无聊",因为无聊的东西能按时上线。

  • 前端:Next.js 15 + TypeScript + Tailwind。画布:Konva.js + perfect-freehand(本期不选 tldraw —— 我们要对记录器钩子有完全的控制权,而且这次也不做多人协作)。
  • 笔输入采集:原生 PointerEvents API。第 1 天就要验证客户驱动是否真的填充了 pressure、tiltX/Y。Wacom 与 Huion 的驱动会;客户的不一定。
  • 笔触标准:使用 Wacom 已公开的 Universal Ink Model(UIM) SDK——既呼应客户 PPT 第 9 页"扩展 WILL / Wintab / HID"的主题,又不必自己发明格式。
  • 后端:FastAPI 部署在 Azure Container Apps。Demo 阶段会话用 SQLite(不上 Postgres——为两周项目折腾过头)。
  • 存储:Demo 机本地的 ./sessions/;若 D9–D10 还有时间,可选接 Azure Blob。
  • Sketch-to-Image 服务:首选 fal.ai(在我们之前的实测中,涂鸦条件模型的 p95 延迟最低);备选 Replicate;企业级备选 Azure OpenAI Image(合规友好,有助于讲好合作故事)。
  • CI / 部署:GitHub Actions;Demo 当天在客户提供的笔记本上本地跑——演示流程不依赖云端。
  • 可观测性:结构化日志 + 一页式仪表板,跟踪笔触采集速率、润色 API 延迟、模型调用成本。
第二周不引入新框架。第 1 天选了什么,就坚持到底。

5 · 为什么两周内不做新的下一笔预测模型

这一节单独成段,因为它最容易被客户质疑。客户 PPT(第 4、7、8、10 页)描述的模型,要吃 (x, y, pressure, tilt, time) 序列、并以"画师本人风格"预测下一笔。在两周内做出来不现实,具体有五个理由:

  1. 目前根本没有带标签的训练数据。这次合作的全部前提就是"市面上没人有过程数据"。一手数据要等到 F2 上线那天才开始产生。要做任何个性化模型,先得积累几周以上的录制。
  2. 公开的笔触模型(Sketch-RNN、StrokeFusion、SwiftSketch)都不吃压感与倾角。它们都把笔触当作 (x, y, pen_state) 三元组。要给它们的架构加 3 个新维度,本质上是研究项目——序列模型的实证工作每个消融实验通常都得几周。
  3. 个性化在我们能"宣称它有效"之前,先得有评估流水线。两周根本无法同时交出评估流水线 + 模型 + UX。
  4. 端到端延迟预算非常紧。客户 PPT 第 7 页承诺 <10 ms 反馈。哪怕一个小型 Transformer 在 200 Hz 采样下也很难压进这个数;我们需要 NPU 级推理与量化。两周内做不到。
  5. "反复试错"恰恰是产品备忘录里点名要避免的事。把 Sprint 算力投在一个不一定产出可运行成果的研究赌注上,是这两周最糟糕的用法。

我们的替代方案:把 F2(记录器 + UIM 导出)定位为客户自己 PPT 第 13 页的 Phase 0:"联合定义标准 & 采集规范"。我们事实上是在为第 13 页 Phase 0 所要的那件交付物买单——同时还给 CEO 准备了一个用现成图像生成"先把模型缺口糊起来"的演示。

时间内容
第 3–6 周(MVP 后)招募约 30 名画师上记录器。积累约 200 小时、带"意图阶段"标签的过程数据。
第 4–8 周第一次模型 Spike:在 (x, y, p, tx, ty, dt) token 流上跑 Transformer。评估下一笔 top-k 准确率,以及"对未见用户"的少样本适配效果。
第 3–6 个月客户路线图里真正的 Phase 1:把"笔迹预测 + 透视辅助"产品化上线。
这个排期既兑现了团队"R&D 周期更长"的现实判断,又落在客户 PPT 第 13 页所规定的"Phase 1(3–6 个月)"窗口里——也就是说,我们没有拖延他们的计划,我们只是在 Phase 0 里把前 14 天重新排了顺序。

6 · 两周窗口的风险

风险可能性缓解措施
客户的笔驱动没有把 tiltX/Y 暴露给浏览器中第 1 天就在真实硬件上验证。回退方案 A:写一个原生小助手程序(Electron / Win32),读取 Wintab 后通过 WebSocket 转发。回退方案 B:优雅降级——记录已有字段,在 .uim 中把 tilt 标为 null。
Demo 当天 Sketch-to-Image 服务挂掉低–中第 9 天预先缓存 4 张涂鸦 × 4 种风格的结果。Demo 时可平滑回退到缓存回放,观众觉察不出。
Demo 机器上的 PointerEvents 采样率过低(<60 Hz)中使用 getCoalescedEvents() 取回子帧采样(Wacom 驱动据此可达 200+ Hz)。第 1 天验证。
无节制的图像 API 调用导致成本飙升低每会话上限 + 每日金额上限 + 后台一键关停开关。
范围蔓延 ——"能不能再加多人/语音/课程?"高就靠这份备忘录。第 0 天就拿到签字。
预期错位 —— CEO 看完润色 Demo 后问"那我们的'下一笔预测'呢?"高Demo 脚本必须先讲 F3(透视/对称/回放),明确点题:"这就是凭你们的笔数据、不需要任何新模型,今天就能做到的事。"然后再讲 F1 作为润色外壳。F4 仅在已上线时演示,并坦白说明它是公开模型,不是"你们专属"的。

7 · 向客户提出的开放问题(在第 0 天提出)

  1. Demo 应面向哪一个数位屏 SKU?确认操作系统(Windows?像 Wacom Bamboo 那样走 USB-HID?还是他们自家的驱动?),以及是否有 Linux/macOS 版本。
  2. 对未来的端侧推理运行时有偏好吗——ONNX Runtime、TensorRT、高通 SNPE、联发科 APU?这影响 Phase 1 的模型选型,但不影响两周 MVP。
  3. 他们的驱动团队能否给我们原始 HID 报文的访问权限(可能有更高采样率/额外字节),还是只能拿到 OS 层的 (x, y, pressure, tilt)?这才是真正决定护城河深度的问题。
  4. Demo 调用第三方图像生成 API(fal/Replicate/Azure OpenAI),还是必须自托管?这是合规驱动的决策,直接影响选型。
  5. Demo 在哪一个品牌名下露脸——他们的、我们的,还是中立的"DrawCopilot Lab"?

8 · 需要的决策签字

第 0 天之前,需要产品/管理层签字三项:

  • 范围 —— 批准 F1 + F2 + F3 这一组合,并明确批准推迟"下一笔预测新模型"。
  • 供应商 —— 批准 Sketch-to-Image 选型(推荐 fal.ai,Azure OpenAI 作为面向企业合规的备选)。
  • Demo 叙事 —— 批准脚本里"护城河在我们刚刚开始记录的那批数据里,而不是 Demo 一上来调用的那个润色 API"这一主线。这是最关键的一项叙事决策:它决定 CEO 离场时是被 AutoDraw(同质化)打动,还是被"未来 6 个月里只有他们能做出来的东西"(真正的合作论点)打动。
